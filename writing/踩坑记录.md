## 3. 评测与指标

| ID | 日期 | 问题 | 客观现象 | 经验与处理 | 状态 |
|---|---|---|---|---|---|
| PIT-E01 | 2026-07-10 | 没设置 seed 的论文代码可以用一次运行复现 | 原 Novelty 协议使用 `temp=1`，论文靠多次 rollout 的均值处理波动 | 复现论文时保留其采样协议；建立本地基线时另设固定 seed，并明确两种协议不可混比 | `已处理` |
| PIT-E02 | 2026-07-14~16 | Accuracy 很低就代表模型不会该任务 | Meta Reviewer 上 DeepSeek 格式有效率仅 4.75%，SciLitLLM 为 0%；近零 Acc 被输出协议失配严重污染 | 主指标旁必须报告格式有效率；格式率异常时先修解析 / prompt，再讨论能力 | `已识别` |
| PIT-E03 | 2026-07-15 | 只按 Accuracy 排名就能代表复杂多分类任务的优劣 | Meta Reviewer 中 Gemma Acc 最高为 0.2311，但 macro-F1 仅 0.0581；类别和预测分布明显不均衡 | 多分类至少同时报告 Acc、macro-F1、逐类召回、格式率；三轴任务另报各轴分母和 Acc | `已识别` |
| PIT-E04 | 2026-07-20~21 | `temp=0` 就意味着同模型同数据结果完全一致 | 同一 DeepSeek checkpoint-410 的两份 coherence 记录为 0.7300 和 0.7490；正式 Novelty 的部分 `temp=0` rollout 也有非零 std | 暂不假定确定性；对比预测差异，并记录模型 hash、merge hash、vLLM 版本和确定性设置 | `根因未确认` |
| PIT-E05 | 2026-07-20 | 目录名和 `exp_name` 足以确认评测使用的 checkpoint | `...ckpt410/metrics.json` 内 `exp_name` 写成 ckpt205；`...ckpt615/metrics.json` 实际 adapter 仍是 checkpoint-410 | 结论以 metrics 内 adapter 路径为准；运行前后断言目录名、`exp_name`、adapter step 三者一致 | `待重跑` |
| PIT-E06 | 2026-07-20 | 不同训练数据上的 validation loss 可以直接比较教师质量 | DeepSeek 和 GLM 的接受样本、划分 ID、目标推理文本都不同 | 教师比较使用共同样本 ID、相同步数和同一外部测试集；训练 loss 只用于同一 run 内选 checkpoint | `待公平对照` |
| PIT-E07 | 2026-07-20~21 | 测试样本相同，就可以直接计算微调增益 | coherence 的 1,048 条 prompt / 标签相同，但 base 使用 16k / 2,048 输出上限，LoRA 使用 8k / 512 | 现有 pp 只作描述；base 和 adapter 必须由同一评测入口、长度上限和模型模板重跑后再做结论 | `待重跑 base` |
