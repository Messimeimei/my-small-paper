# 写作思路

## 1. 最终研究定位

本文研究的不是“把同一条评价过程拆成多个阶段并分别训练”，而是：

**在科学文献场景中，定义若干输入输出形式不同、能够独立训练的基础评价任务，为每个任务训练一个 LoRA 专家，再组合这些专家解决训练阶段未见过的复杂科学评价任务。**

核心研究问题是：

> 以可复用基础任务为单位训练 LoRA 专家，是否比单一多任务微调、按原始数据集训练专家或随机划分专家，更有利于复杂科学评价任务的组合迁移？

这与 MoDE-CoTD 的关系是：

- MoDE-CoTD 提供“冻结主干、分别训练任务 LoRA、推理时组合”的技术框架；
- 本文不创新通用 MoE 结构，而是在科学评价领域研究“应该用什么基础任务训练专家”；
- 本文的主要创新变量是**基础任务定义与组合迁移实验**，而不是路由器本身。

如果只是把 Related Work 和 RevUtil 各训练一个 LoRA，再在其他数据集上加权，属于直接应用 MoDE-CoTD，论文创新性较弱。只有证明“按自己定义的基础评价任务训练专家”优于“按原始数据集训练专家”，研究贡献才成立。

## 2. 任务定义

### 2.1 科学评价任务

本文将科学评价任务定义为：

**给定科学文本、评价对象和评价标准，模型需要获取相关证据，判断证据与待评对象之间的关系，并依据 rubric 输出规范性结论。**

形式上可写为：

`y = f(x, q, r)`

其中：

- `x`：论文、摘要、相关工作、评审意见或修订文本；
- `q`：待评价的主张、修改、评论或关系；
- `r`：评价维度和评分细则；
- `y`：证据、关系标签、等级分数或最终类别。

G-Eval [R11] 和 Prometheus [R12] 说明评价行为依赖评价标准和评分过程；SciRM [R2] 说明这种 rubric 驱动的评价形式适用于科学写作场景。

### 2.2 什么是基础任务，什么是复杂任务

本文所说的“基础任务”必须同时满足：

1. 有独立、明确的输入输出定义；
2. 可以构造独立训练集并单独评测；
3. 对多个复杂目标任务都有潜在作用，而不是只服务一个数据集；
4. 与其他基础任务在操作上不同，不只是输出 JSON 格式不同。

“复杂任务”则定义为：完成最终判断通常需要两个或以上基础任务提供的能力。例如 Scientific Revision 不仅要比较修改前后文本，还要判断修改是否符合指令并保持事实正确；Meta Reviewer 同时需要寻找论文证据、核验评论和论文是否一致、再依据标准分类。

因此，本文不再把任务简单排列为低、中、高难度，而是用“需要哪些基础任务”描述其结构。

## 3. 建议定义的三个基础科学评价任务

第一版只定义三个任务。三个任务分别对应抽取、关系分类和 rubric 评分，数据格式与监督目标真正不同。

### 3.1 基础任务一：科学证据检索

**任务定义**

- 输入：科学文本 `x`、待核验主张或评价问题 `q`；
- 输出：支持判断的原文句子或片段；如果没有相关证据，则输出 `NO_EVIDENCE`；
- 任务形式：抽取式生成或句子排序。

示例：

```text
输入：论文摘要/正文 + “该方法是否进行了跨数据集实验？”
输出：与跨数据集实验直接相关的原文句子，或 NO_EVIDENCE。
```

**可用数据来源**

- SciFact [R14] 的科学主张与证据句标注；
- Related Work 训练数据中引用内容与定位判断所依赖的文本；
- RevUtil 原始训练数据中可追溯到论文正文的评论和证据位置；
- 若后两项没有原文证据位置，只用于任务二或任务三，不能伪造成抽取监督。

**专家作用**

该 LoRA 专家学习从长科学文本中找到与当前判断相关的局部证据，为 Revision 和 Meta Reviewer 等长文本评价提供依据。

### 3.2 基础任务二：科学主张—证据关系核验

**任务定义**

- 输入：待核验主张 `q`、候选证据 `e`；
- 输出：`SUPPORTS / CONTRADICTS / INSUFFICIENT`；
- 任务形式：三分类关系判断，并可附简短理由。

示例：

```text
输入：评审意见“论文没有消融实验” + 论文中的实验段落
输出：SUPPORTS、CONTRADICTS 或 INSUFFICIENT。
```

**可用数据来源**

- SciFact [R14] 中支持、反驳和证据不足关系；
- Related Work 的 positioning check 可转换为“文本是否支持论文定位”的关系核验；
- RevUtil 的 verifiability、grounding specificity 可用于构造主张与依据关系；
- 可从同领域论文中交换证据生成 hard negative，但必须人工抽检，避免负样本只靠主题差异即可判断。

**专家作用**

该 LoRA 专家学习比较两个科学文本单元之间的语义关系，是 Novelty Alignment、Revision correctness 和 Meta Reviewer correctness 的基础。

### 3.3 基础任务三：基于 rubric 的科学评价打分

**任务定义**

- 输入：待评价文本 `q`、显式 rubric `r`，必要时附证据状态；
- 输出：rubric 中规定的二元标签、等级分数或类别；
- 任务形式：标准驱动分类或序数评分。

示例：

```text
输入：一条审稿意见 + actionability 的 1-5 分标准
输出：分数 + 对应的标准条目。
```

**可用数据来源**

- RevUtil 中 actionability、helpfulness、grounding specificity、verifiability；
- Related Work 中 coherence、positioning type、positioning check；
- 这些数据必须使用官方训练部分或重新划分的训练部分，当前测试文件不能直接用于训练后继续报告官方测试成绩。

**专家作用**

该 LoRA 专家学习依据不同 rubric 做规范性判断，负责把前两个专家得到的证据状态和关系状态映射为最终评价标签。

### 3.4 为什么暂时只做三个任务

这三个任务构成一条清楚但不强制串行的评价链：

`检索证据 -> 核验证据关系 -> 依据 rubric 判断`

它们具有独立数据、独立指标和不同输出形式，适合分别训练 LoRA。暂不增加“不确定性估计”“总结”“规划”等专家，因为当前没有稳定监督；专家数量过多还会增加参数匹配和组合搜索成本。

这里的“基础任务”是本文提出的一种工程化任务分解，不应声称是人类或模型内部真实、不可再分的推理原子。

## 4. 复杂目标任务如何由专家组合

在看到目标测试结果之前，依据任务说明和 rubric 预先定义任务—专家需求矩阵：

| 复杂目标任务 | 证据检索专家 | 关系核验专家 | Rubric评分专家 | 实验角色 |
|---|---:|---:|---:|---|
| Related Work | 可选 | 需要 | 需要 | seen / 训练域测试 |
| RevUtil | 可选 | 可选 | 需要 | seen / 训练域测试 |
| Novelty Alignment | 不需要或低权重 | 需要 | 需要 | 小样本 unseen 辅助测试 |
| Scientific Revision | 需要 | 需要 | 需要 | 主要 unseen 测试 |
| Meta Reviewer | 需要 | 需要 | 需要 | 外部压力测试 |

这张表是可检验的先验，不是实验后根据结果解释出来的。正式实验前应冻结版本，并由两名标注者根据任务说明独立判断每个任务需要哪些专家。

需要特别注意：复杂任务并不一定要求严格执行三次独立推理调用。本文主方法仍然可以像 MoDE-CoTD 一样，在参数空间组合多个 LoRA 更新，再由组合后的模型一次生成结果。上述链条用于定义专家能力和解释组合，不等同于固定流水线。

## 5. 核心假设与研究问题

### 5.1 可证伪假设

**H1：基础任务专门化假设。** 每个 LoRA 专家应在自己对应的基础任务测试集上明显优于其他单专家和基础模型。

**H2：组合增益假设。** 在复杂未见任务上，预先指定的多专家组合应优于最佳单专家，而不是只有某一个专家决定全部结果。

**H3：任务定义有效性假设。** 按“证据检索、关系核验、rubric 评分”划分的专家，应优于按原始数据集划分和随机划分的专家。

**H4：路由一致性假设。** 学习路由器在未见任务上的权重应与预先冻结的任务—专家需求矩阵大致一致，并且错误组合会产生可预测下降。

### 5.2 研究问题

- **RQ1：三个基础任务能否训练出功能不同的科学评价 LoRA 专家？**
- **RQ2：组合基础任务专家是否能够提升未见复杂科学评价任务？**
- **RQ3：提升是否来自本文的基础任务定义，而不是 CoT、多专家容量或随机集成？**
- **RQ4：固定组合与学习路由相比，哪一种在未见任务上更稳定？**

不再把“缓解通用推理退化”作为主研究问题。若要研究遗忘，需要额外通用推理基准和完整微调对照，会分散本文主线。

## 6. 数据使用方案

### 6.1 当前数据状态

当前仓库中的五个 `prompted_*_data.json` 文件均只有测试样本，`train` 为空：

| 文件 | 样本数 | 用途 |
|---|---:|---|
| `prompted_rw_gen_data.json` | 1857 | seen 测试 |
| `prompted_rev_util_data.json` | 4788 | seen 测试 |
| `prompted_novelty_data.json` | 76 | 小样本 unseen 辅助测试 |
| `prompted_revision_data.json` | 6184 | 主要 unseen 测试 |
| `prompted_meta_reviewer_data.json` | 779 | 外部压力测试 |

这些测试文件可用于基线复现和最终评测，但不能用于定义专家、训练路由或调超参数。

### 6.2 需要建立的三个训练集

最终训练数据不按目标 benchmark 名称简单合并，而按三个基础任务重新组织：

| 训练集 | 主要来源 | 输出形式 | 对应专家 |
|---|---|---|---|
| Evidence Retrieval Train | SciFact + 有证据位置的科学评价训练数据 | 证据句/NO_EVIDENCE | 检索专家 |
| Relation Verification Train | SciFact + 定位/grounding关系样本 + hard negatives | 三分类 | 核验专家 |
| Rubric Evaluation Train | Related Work train + RevUtil train | 二分类/1-5分 | 评分专家 |

首选方案是获取 SciRM 的官方 Related Work 和 RevUtil 训练数据，并引入 SciFact 的官方训练划分。所有来源按论文 ID、claim ID 或文档 ID 做 group split，避免同一论文内容跨训练与测试。

若无法获得 SciRM 官方训练部分，只能从当前 Related Work 和 RevUtil 数据重新建立 group split。此时必须放弃继续把完整数据结果称为官方 test 成绩，并公开新的划分。

### 6.3 数据构造必须控制的风险

- 证据检索任务必须有真实 span 或句子 ID，不能只用最终分类标签伪造抽取监督；
- hard negative 要尽量保持主题相似，否则关系专家可能只学主题匹配；
- 教师模型生成的数据必须限制在训练语料，并抽取每类至少 100 至 200 条人工核验；
- Novelty、Revision 和 Meta Reviewer 的标签不得用于专家训练、路由训练和超参数选择；
- 若使用目标任务的少量标签学习组合权重，必须另列为 few-shot adaptation，不能称为 zero-shot transfer。

## 7. 方法设计

### 7.1 专家训练

固定同一个基础模型，分别在三个基础任务训练：

- `LoRA_retrieval`：证据句抽取或排序；
- `LoRA_verification`：主张—证据三分类；
- `LoRA_rubric`：依据评价标准打分。

三个专家使用各自原生任务格式进行完整 SFT 或 CoT 蒸馏，不再使用“同一条轨迹不同 token 区域的掩码损失”。为了降低纯输出格式造成的干扰，可以统一外层格式：

```text
<reasoning>...</reasoning>
<answer>任务自己的证据、关系标签或分数</answer>
```

这样输入结构和监督任务保持不同，但模型不会因为 XML/JSON 包装差异形成虚假的专家专门化。

### 7.2 专家组合

冻结三个专家，按 MoDE-CoTD 的方式组合 LoRA 更新：

`Delta W(x, r) = w_r Delta W_retrieval + w_v Delta W_verification + w_s Delta W_rubric`

比较两种组合策略：

1. **固定任务组合**：根据第 4 节预先冻结的任务—专家矩阵设置权重；这是检验“专家是否可组合”的主要、最容易解释的设置。
2. **学习路由组合**：路由器根据目标 instruction、rubric 和输入生成权重；路由器只能用基础任务训练集及其合成组合样本训练。

如果路由器只见过“一个样本对应一个专家”的 one-hot 监督，它没有理由在未见复杂任务上自动输出多专家组合。因此需要从基础任务训练数据构造双任务/三任务组合 episode，或使用预定义多标签需求监督训练路由器。固定组合应先于复杂路由完成。

### 7.3 参数和训练预算匹配

所有可训练方法必须保持：

- 相同基础模型；
- 相同总训练样本和训练 token；
- 相同总 LoRA rank 或可训练参数量；
- 相同单次推理激活参数量；
- 相同教师模型和推理轨迹质量；
- 相同最大上下文和解码设置。

第一轮建议使用 `Qwen3-4B` 完成全部对照。资源允许时，再使用 `Qwen2.5-7B-Instruct` 验证方法是否依赖单一底座。

## 8. 必须完成的实验

### 8.1 实验一：基础任务专家是否训练成功

分别在三个基础任务的独立测试集上比较：

- Base Model；
- 单一多任务 LoRA；
- 三个单独 LoRA 专家；
- 随机划分专家。

应得到一个专家 × 基础任务矩阵。支持 H1 的理想结果是明显的对角优势：检索专家最擅长证据检索，核验专家最擅长关系分类，评分专家最擅长 rubric 评价。

如果三个专家在三个任务上的表现没有明显差异，就不能声称训练出了不同基础能力，后续组合结果只能解释为普通模型集成。

### 8.2 实验二：复杂任务上的组合增益

对每个复杂任务完整评测以下组合：

- Base；
- 每个单专家，共 3 组；
- 每两个专家组合，共 3 组；
- 三专家固定组合；
- 三专家均匀组合；
- 三专家学习路由组合。

这个实验直接回答“拼起来是否比单个专家更好”。仅比较 Base 和完整方法是不够的。

主要证据来自 Scientific Revision。Novelty 只有 76 条，只能作为方向一致的辅助证据；Meta Reviewer 是经过重构的长文本压力测试，不能和其他任务直接平均。

支持 H2 的最低要求是：预定义组合在 Revision 上稳定优于最佳单专家，并且在 Novelty 或 Meta Reviewer 至少一个额外任务上方向一致。

### 8.3 实验三：你的任务定义是否真的更好

必须比较四种划分：

1. **Single Multi-task LoRA**：三个基础任务合并训练一个 LoRA；
2. **Dataset-level MoDE**：按 SciFact、Related Work、RevUtil 等原始数据集训练专家；
3. **Random-partition MoDE**：相同数据随机分为三个专家；
4. **Primitive-task MoDE**：按检索、核验、rubric 评分训练专家，即本文方法。

四组使用相同专家数量、总 rank、训练 token 和路由方式。这个实验是整篇论文最重要的创新性证据：它检验的是“自己定义基础任务有没有价值”，而不是“MoDE 是否有效”。

### 8.4 实验四：消融和反证

- 删除预期需要的专家与删除无关专家；
- 打乱任务—专家对应关系；
- 将 hard negative 换成随机负样本；
- 去掉 CoT，只保留最终标签；
- 固定组合与学习路由对比；
- 路由器不使用合成组合 episode；
- 单 LoRA 与多个 LoRA 参数量匹配；
- 错误组合权重测试。

最有说服力的现象是：Revision 移除关系核验或 rubric 专家下降明显，Novelty 移除关系核验专家下降明显，而移除低权重检索专家的影响较小。该结果只能说明模型行为与任务分解一致，不能证明模型内部存在真实推理原子。

### 8.5 统计规范

- 每个训练方法至少 3 个随机种子；
- 使用 paired bootstrap 报告置信区间；
- 同一样本预测使用 McNemar 或 permutation test；
- Novelty 报告 exact interval，避免过度解释几个样本的变化；
- 类别不平衡任务报告 macro-F1 和 majority baseline；
- 不同标签空间的原始 accuracy 不直接求平均；
- 确定性单次推理不再写成 `mean ± 0.0000`，训练种子方差和生成采样方差分开报告。

## 9. 当前五个评价数据集如何充分利用

### Related Work

- 官方训练部分主要用于 rubric 评分任务；
- 有原文对应关系时，可补充证据检索和关系核验训练；
- 测试部分用于检查专家组合在训练域任务上是否退化。

### RevUtil

- actionability、helpfulness、grounding specificity、verifiability 用于 rubric 评分训练；
- verifiability extraction 可用于构造“是否包含待核验主张”的辅助关系任务；
- 如果没有论文正文，不能把评论强行用于证据 span 检索。

### Scientific Revision

- 作为主要 unseen 复杂任务；
- relatedness 更偏指令—修改关系核验；
- correctness 同时依赖关系核验和 rubric 评分；
- 完全不参与专家和路由训练。

### Novelty Alignment

- 作为关系核验 + rubric 评分组合的辅助测试；
- 只有 76 条，不能作为唯一主结果；
- 重点比较最佳单专家与双专家组合，而不只是完整模型与 Base。

### Meta Reviewer

- 作为检索 + 核验 + rubric 评分三专家组合的外部压力测试；
- 10 类结果以 macro-F1 为主，同时报告三项 cascade 指标；
- 明确说明数据经过长度筛选和任务重构，不把它称为标准 benchmark 复现。

## 10. 论文写作逻辑

### 10.1 引言逻辑

1. 科学评价要求依据 rubric 对科学文本和证据关系做判断 [R2][R11][R12]；
2. 现有科学领域适配和 SciRM 已证明领域训练与多任务训练有效，但主要学习整体任务 [R1][R2]；
3. MoDE-CoTD、Self-MoE 和 LoraHub 说明多个 LoRA 能够被组合 [R7][R8][R9]；
4. 但现有工作没有研究在科学评价领域，哪些基础任务最适合作为可组合专家的训练单位；
5. 因此本文定义证据检索、关系核验和 rubric 评分三个基础任务，并比较基础任务划分、数据集划分和随机划分在复杂未见任务上的迁移。

### 10.2 贡献表述

实验完成前，贡献写成：

1. 提出一组面向科学评价的基础任务定义及相应训练数据组织方式；
2. 基于 MoDE-CoTD 构建基础任务 LoRA 专家组合方法；
3. 通过最佳单专家、数据集专家、随机专家和多任务 LoRA 对照，系统检验基础任务组合能否迁移到未见复杂评价任务。

实验完成后，只有在结果支持时才能写“基础任务专家组合显著优于任务级或随机组合”。

### 10.3 推荐标题

- **基于基础任务专家组合的复杂科学评价方法研究**
- **面向未见科学评价任务的 LoRA 专家组合研究**
- **科学评价中基础任务定义与模块化迁移研究**

第一个最符合当前方法；第三个更强调真正的研究变量。

## 11. 文献支撑及作用边界

### [R1] SciLitLLM

- 来源：https://arxiv.org/abs/2408.15545
- 内容：通过持续预训练和监督微调增强科学文献理解。
- 作用：说明科学领域适配的必要性。
- 边界：不能证明基础评价任务可组合。

### [R2] Reward Modeling for Scientific Writing Evaluation

- 来源：https://arxiv.org/abs/2601.11374
- 本地笔记：[Reward Modeling for Scientific Writing Evaluation（SciRM & SciRM-Ref）.md](/data01/public/yangxin/small-paper/paper/面向科学文献跨学科推理的模型微调方法研究/笔记/Reward%20Modeling%20for%20Scientific%20Writing%20Evaluation（SciRM%20&%20SciRM-Ref）.md)
- 内容：提出 SciRM / SciRM-Ref，依据显式 rubric 进行多维科学写作评价并测试未见任务。
- 作用：提供主要目标任务、评价数据和统一模型基线。
- 边界：它没有比较不同专家任务定义。

### [R3] SciGPT

- 本地笔记：[SciGPT.md](/data01/public/yangxin/small-paper/paper/面向科学文献跨学科推理的模型微调方法研究/笔记/SciGPT.md)
- 内容：强调科学专用模型、跨学科知识和长文本能力。
- 作用：补充科学领域模型背景。
- 边界：与专家组合关系较弱。

### [R4] On the Impact of Fine-Tuning on Chain-of-Thought Reasoning

- 来源：https://arxiv.org/abs/2411.15382
- 内容：研究微调对 CoT 和 OOD 泛化的影响。
- 作用：作为普通微调可能产生能力干扰的次要背景。
- 边界：本文不做通用推理保持实验时，不能作为核心动机。

### [R5] Catastrophic Forgetting During Continual Fine-tuning

- 来源：https://arxiv.org/abs/2308.08747
- 内容：研究持续微调中的知识和能力遗忘。
- 作用：补充模块化适配的背景。
- 边界：本文不是持续学习设置。

### [R6] Chain-of-Thought Prompting

- 来源：https://arxiv.org/abs/2201.11903
- 内容：说明显式中间步骤可帮助部分复杂推理任务。
- 作用：支持为各基础任务构造解释或 CoT 监督。
- 边界：不能证明 CoT 是真实推理过程。

### [R7] MoDE-CoTD

- 本地论文：[MoDE-CoTD Chain-of-Thought Distillation for Complex Reasoning.pdf](/data01/public/yangxin/small-paper/paper/垂域微调与推理能力/论文/MoDE-CoTD%20Chain-of-Thought%20Distillation%20for%20Complex%20Reasoning.pdf)
- 本地笔记：[MoDE-CoTD.md](/data01/public/yangxin/small-paper/paper/垂域微调与推理能力/笔记/MoDE-CoTD.md)
- 内容：冻结主干，在不同推理任务上训练 LoRA 专家并动态组合。
- 作用：本文的直接技术基础和 dataset/task-level 专家基线。
- 边界：本文必须证明科学评价基础任务定义带来额外价值，而不是直接复现。

### [R8] Self-MoE

- 来源：https://arxiv.org/abs/2406.12034
- 内容：利用自生成数据训练多个自专业化专家并动态路由。
- 作用：支持自构造任务数据和动态路由的可行性。
- 边界：第一版不建议同时完整引入其自生成流程，避免多个创新变量混杂。

### [R9] LoraHub

- 来源：https://arxiv.org/abs/2307.13269
- 内容：组合已有任务 LoRA 以适配未见任务。
- 作用：提供 LoRA 组合及未见任务迁移基线。
- 边界：不能证明本文的基础任务划分最优。

### [R10] Cell-o1

- 来源：https://arxiv.org/abs/2506.02911
- 内容：在单细胞领域通过显式推理训练提升专业任务表现。
- 作用：说明垂直科学任务可以受益于推理训练。
- 边界：任务形式与科学评价不同，只作补充案例。

### [R11] G-Eval

- 来源：https://arxiv.org/abs/2303.16634
- 内容：把生成评价表示为评价标准、评分步骤和结构化输出。
- 作用：支撑 rubric 评分任务的定义。
- 边界：不研究科学评价专家组合。

### [R12] Prometheus

- 来源：https://arxiv.org/abs/2310.08491
- 内容：训练依据用户 rubric 做细粒度评价的模型。
- 作用：支撑 rubric 评价是一项可独立训练的任务。
- 边界：不证明本文三任务分解最优。

### [R13] Task Arithmetic

- 来源：https://arxiv.org/abs/2212.04089
- 内容：研究不同任务参数更新的组合与干扰。
- 作用：说明能力组合和组合单位需要实验验证。
- 边界：不是 LoRA 路由设置。

### [R14] SciFact: A Dataset for Verifying Scientific Claims

- 来源：https://arxiv.org/abs/2004.14974
- 内容：提供科学主张、相关论文摘要、证据句和支持/反驳标签。
- 作用：为证据检索专家和主张—证据关系核验专家提供独立、可追溯的监督。
- 边界：它是科学事实核验，不是科学写作 rubric 评价，因此需要与 SciRM 训练数据组合使用。

## 12. 实际执行顺序

1. 获取 SciRM 的 Related Work / RevUtil 训练数据和 SciFact 数据，完成许可证与划分检查。
2. 分别实现证据检索、关系核验、rubric 评分三个基础任务的数据转换器和独立评测。
3. 先训练三个单专家，确认专家 × 基础任务矩阵具有对角优势。
4. 实现固定 LoRA 组合，完整跑单专家、双专家和三专家组合；先不要训练复杂路由。
5. 完成 Single LoRA、Dataset-level MoDE、Random MoDE、Primitive-task MoDE 四组关键对照。
6. 固定所有设置后，以 Revision 为主要 unseen 结果，Novelty 为辅助结果，Meta Reviewer 为压力测试。
7. 最后再训练 learned router；如果固定组合已经无增益，不应通过继续增加路由复杂度掩盖基础任务定义无效的问题。
