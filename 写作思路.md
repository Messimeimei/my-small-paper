# 写作思路

## 1. 严格审视后的研究定位

本文不再预设“科学评价能力天然由若干真实推理原子组成”，也不把目标写成“使用 MoDE-CoTD 或 Self-MoE 提升数据集准确率”。更稳妥、可验证的研究问题是：

**在科学评价任务中，将训练过程按可解释的评价技能组件进行模块化，是否比按任务划分或随机划分专家，更有利于未见评价任务的迁移？**

这里使用“评价技能组件（evaluation skill components）”，暂不使用“推理原子”。原因是现有数据只能支持对功能性组件进行操作化和实验验证，不能证明这些组件是模型内部真实、不可再分的认知单元。如果实验同时证明组件具有选择性、可组合性和跨任务复用性，再在讨论中将其称为“近似推理原子”。

本文的最小创新点不是新的 MoE 结构，而是：

1. 提出一种面向科学评价任务的技能级专家划分方式；
2. 在相同数据、参数量和路由结构下，比较技能划分、任务划分与随机划分；
3. 通过未见任务、随机专家和选择性消融检验性能提升是否确实来自技能划分。

## 2. 任务定义与研究边界

### 2.1 科学评价任务的操作化定义

本文将科学评价任务定义为：

**给定科学文本、评价对象和显式评分细则，模型需要定位相关证据，将证据与评价标准对应，并输出规范性判断。**

形式上，单个样本可写为：

`y = f(x, r)`

其中，`x` 是论文、修订文本、相关工作或评审意见，`r` 是评价维度及其评分细则，`y` 是标签或分数。与普通抽取任务相比，评价结果不仅取决于文本内容，还取决于评分细则；同一文本在不同标准下可能得到不同判断。

G-Eval [R11] 和 Prometheus [R12] 支撑“评价对象 + rubric + 评价过程”的任务定义；SciRM [R2] 则说明这一形式在科学写作评价中具有直接应用。

### 2.2 不再使用未经验证的低、中、高复杂度排序

Related Work、RevUtil、Novelty Alignment、Scientific Revision 和 Meta Reviewer 不直接被排列为低、中、高难度。原始准确率还受到标签分布、样本规模、文本长度、提示模板和输出空间影响，不能等同于推理复杂度。

本文只描述每项任务的**技能需求画像**：

| 任务 | 证据分析 | 标准应用 | 一致性核验 | 在实验中的角色 |
|---|---:|---:|---:|---|
| Related Work | 中 | 高 | 中 | 训练任务 + seen 测试 |
| RevUtil | 高 | 高 | 中 | 训练任务 + seen 测试 |
| Novelty Alignment | 中 | 高 | 高 | 小样本 unseen 辅助测试 |
| Scientific Revision | 高 | 高 | 高 | 主要 unseen 测试 |
| Meta Reviewer | 高 | 高 | 高 | 外部压力测试 |

这张表是待验证的任务分析，不是实验结论。正式论文中应由两名标注者依据任务说明和 rubric 独立标注，并报告一致率及分歧处理过程。由于任务数较少，不应只对五行任务级标签计算 kappa；更有意义的是抽取训练样本和诊断样本做样本级组件标注。

### 2.3 三个评价技能组件

为保证当前数据能够提供监督，第一版只保留三个组件：

1. **证据分析（Evidence Analysis）**
   - 从输入中定位与评价维度有关的事实、主张或修改内容；
   - 判断证据是否存在、是否具体、是否与评价对象相关；
   - 主要由 `positioning_check`、`grounding_specificity`、`verifiability_extraction` 等维度提供监督。

2. **标准应用（Criterion Application）**
   - 解释 rubric 中每个等级或标签的含义；
   - 将证据状态映射到 actionability、helpfulness、coherence、verifiability 等评价等级；
   - 重点学习“为什么该证据满足或不满足某一标准”。

3. **一致性核验（Consistency Verification）**
   - 检查文本、证据、评价标准和候选结论之间是否一致；
   - 包括主张与证据一致性、修改前后的一致性以及两种评价结论的一致性；
   - seen 数据中可从定位核验、证据与结论核验等样本构造监督，Novelty 和 Revision 只用于测试其组合迁移能力。

暂时删除“不确定性估计”组件。现有训练数据没有稳定的不确定性标签，强行设置该专家会造成只有名称、没有独立监督的问题。

## 3. 核心假设与研究问题

### 3.1 可证伪假设

**H1：技能专门化假设。** 如果上述划分具有实际语义，单个技能专家应在对应的独立诊断集上优于其他技能专家和随机专家。

**H2：组合迁移假设。** 在训练数据、专家数量、总 LoRA rank 和路由方式匹配时，技能级专家组合应比任务级专家和随机划分专家在未见任务上获得更稳定的增益。

**H3：结构一致性假设。** 关闭某项任务所需的技能专家，造成的下降应显著大于关闭无关专家；路由权重也应与预先标注的技能需求画像一致。

### 3.2 研究问题

- **RQ1：所定义的评价技能组件能否形成可区分的专家能力，而不只是数据集风格的不同名称？**
- **RQ2：技能级专家是否比多任务 LoRA、任务级专家和随机专家更有利于未见科学评价任务迁移？**
- **RQ3：未见任务上的增益是否与技能需求匹配和专家组合有关，而不是仅由参数量、CoT 数据或 MoE 结构带来？**

原来的“是否缓解通用推理退化”不再作为主研究问题。证明能力保持需要额外的通用推理基准和完整微调对照，会使小论文同时承担两条独立主线。若资源充足，可把它作为补充实验，而不是核心贡献。

## 4. 当前数据如何使用

### 4.1 当前仓库中的数据状态

当前 `eval_data/prompted_*_data.json` 五个文件的 `train` 均为空，只有测试部分：

| 文件 | 样本数 | 用途 |
|---|---:|---|
| `prompted_rw_gen_data.json` | 1857 | seen 测试；不能直接当作完整训练集继续报告原测试成绩 |
| `prompted_rev_util_data.json` | 4788 | seen 测试；不能直接当作完整训练集继续报告原测试成绩 |
| `prompted_novelty_data.json` | 76 | 小样本 unseen 辅助证据 |
| `prompted_revision_data.json` | 6184 | 主要 unseen 测试 |
| `prompted_meta_reviewer_data.json` | 779 | 外部压力测试 |

这些文件以及已有模型输出可以用于基线复现、错误分析、标签分布分析和最后评测，但不应用于选择技能组件、调路由、调超参数或生成训练轨迹。

### 4.2 首选训练数据方案

必须优先获取 SciRM 中 Related Work 和 RevUtil 的官方训练集或其可追溯原始训练数据，并按原始论文、作者或文档 ID 做 group split：

- 训练集：训练三个技能专家、任务专家和全部基线；
- 开发集：早停、超参数选择和路由训练；
- 当前 prompted test：只在方案冻结后运行一次主评测。

如果官方训练数据无法获得，备用方案是从当前 Related Work 和 RevUtil 数据中重新建立固定的 `70/15/15` group split。此时必须：

- 放弃把原完整测试结果作为可比的官方 test 成绩；
- 发布新的划分 ID 和随机种子；
- 保证同一论文、同一来源段落或近重复样本不跨 split；
- Novelty、Revision 和 Meta Reviewer 仍保持完全不参与训练。

不允许随机按行切分，因为同一论文的不同片段可能同时进入训练和测试，产生内容泄漏。

### 4.3 五个数据集的具体角色

**Related Work 和 RevUtil：训练与技能诊断来源。**

- 训练统一多任务 LoRA、任务级专家和技能级专家；
- 从训练部分抽取少量样本构建人工核验的技能诊断集；
- 测试部分只报告 seen 性能，验证模块化没有明显牺牲已见任务。

**Scientific Revision：主要迁移证据。**

- 样本量足够，包含 relatedness 和 correctness 两个维度；
- 完全不参与专家训练、路由训练和超参数选择；
- 作为论文中最主要的 unseen 结果。

**Novelty Alignment：辅助迁移证据。**

- 只有 76 条，不能承担核心结论；
- 报告 accuracy、macro-F1、二项分布或 bootstrap 置信区间；
- 使用配对预测做 exact McNemar 或置换检验，不只比较点估计。

**Meta Reviewer：外部压力测试。**

- 当前版本经过长度筛选和任务重构，不能和 SciRM 四项任务直接平均；
- 10 类标签高度不平衡，应以 macro-F1 为主，同时报告 majority baseline；
- 三个 cascade 子维度单独报告；
- 在论文中明确称为 reconstructed external stress test，而不是标准 unseen benchmark。

## 5. 训练数据与技能监督如何构造

### 5.1 结构化评价轨迹

对每条 seen 训练样本，输入由科学文本 `x`、rubric `r` 和金标签 `y` 构成。使用教师模型生成受约束的结构化评价轨迹：

```text
<evidence>与评价维度直接相关的文本证据</evidence>
<criterion>rubric 中适用的判断标准及其解释</criterion>
<verification>证据、标准和候选结论是否一致</verification>
<decision>最终标签及简短理由</decision>
```

教师只能处理 seen 训练样本，不能读取 Novelty、Revision 或 Meta Reviewer 的标签。生成后要执行自动过滤：

- `<evidence>` 必须能够在输入中定位，或明确标记为“未找到”；
- `<criterion>` 必须引用当前 rubric，而不是自造标准；
- `<decision>` 必须与金标签一致；
- 删除格式错误、证据无法定位或理由与标签矛盾的轨迹。

### 5.2 人工质量控制

从每个任务维度至少抽取 30 至 50 条轨迹，由两名标注者检查：

- evidence 是否有文本依据；
- criterion 是否正确解释 rubric；
- verification 是否完成一致性判断；
- decision 是否由前述信息支持。

报告每项通过率、标注一致率和分歧仲裁方式。如果结构化轨迹质量不足，不能直接训练所谓技能专家，否则专家只会蒸馏教师错误。

### 5.3 三类专家的监督形式

三个专家使用相同输入 `x + r` 和相同的完整输出模板，避免训练与组合推理时的格式分布不一致；区别只在于计算损失的 token 区域：

- 证据专家：主要优化 `<evidence>` 区域；
- 标准专家：主要优化 `<criterion>` 区域；
- 核验专家：主要优化 `<verification>` 区域；
- 三个专家都对 `<decision>` 区域施加较小且相同权重的辅助损失，保证每个专家与最终评价输出兼容。

可写为：

`L_k = L_component_k + lambda L_decision`

其中 `lambda` 在 seen 开发集上固定，并对所有专家保持一致。非目标组件区域不计算损失。这样三个专家面对相同数据格式但拥有不同优化目标，避免把完整任务样本换一个名字分别训练，也避免三个异构输出专家在权重组合后无法生成统一答案。应增加一个 `lambda = 0` 的消融，检查辅助决策损失是否掩盖了技能专门化。

## 6. 方法设计

### 6.1 主方法

冻结同一个基础模型，为三个技能分别训练 LoRA 专家。路由器仅根据任务说明、rubric 和当前输入生成三个权重，然后组合 LoRA 更新：

`Delta W(x, r) = w_e Delta W_e + w_c Delta W_c + w_v Delta W_v`

其中 `w_e + w_c + w_v = 1`。路由器只能在 seen 训练/开发数据上学习，不能使用 unseen 任务标签。

训练分为两个阶段，保证从单项技能监督到最终评价输出形成闭环：

1. **专家专门化阶段**：分别使用第 5.3 节的目标训练三个 LoRA 专家；主干冻结，各专家只优化自己的组件目标。
2. **组合决策阶段**：冻结三个专家，仅在 seen 训练数据上优化路由器；目标是生成完整结构化轨迹和正确的 `<decision>`。因此最终标签仍受到监督，但不会再修改专家参数。

开发集只用于选择路由器和早停。对 unseen 任务，路由器直接根据新任务的 instruction、rubric 和输入零样本地产生权重。如果使用 unseen 开发标签调整路由，则必须另行报告为 few-shot adaptation，不能与零样本迁移结果混合。

第一版优先采用软加权组合，避免同时引入 top-k、专家层选择、自生成数据和复杂负载均衡。MoDE-CoTD [R7] 提供“冻结主干 + LoRA 专家组合”的技术模板；Self-MoE [R8] 只作为模块化路由相关工作，不必完整复现其自生成流程。

### 6.2 必须匹配的实验预算

所有可训练基线必须使用：

- 相同基础模型；
- 相同 seen 训练样本；
- 相同训练 token 数或优化步数；
- 相同总 LoRA rank 和可训练参数量；
- 相同教师轨迹版本；
- 相同推理模板和最大上下文。

如果技能专家有三个 rank-8 LoRA，而任务级专家只有两个 rank-8 LoRA，应调整 rank 或激活预算，使总参数量与单次推理激活参数量都可比较。

### 6.3 基础模型选择

第一轮建议只使用一个基础模型完成全部对照。结合现有基线，优先选择 `Qwen3-4B` 作为主模型；它在 4B 量级上已有较完整的五任务结果。资源允许时，再用 `Qwen2.5-7B-Instruct` 做一次跨模型复验。不同基础模型的结果分开报告，不能把模型规模差异混入方法增益。

## 7. 完整实验设计

### 7.1 基线与控制组

1. **Base Model**：不训练。
2. **Label-only Multi-task LoRA**：只使用最终标签，检验普通任务微调。
3. **CoT Multi-task LoRA**：使用完整结构化轨迹，控制提升是否只来自 CoT 蒸馏。
4. **Task-level Experts**：分别按 Related Work / RevUtil 训练完整评价轨迹专家，使用同类路由器，并通过 LoRA rank 匹配总参数量和激活参数量。
5. **Random-partition Experts**：将组件训练实例随机分配给三个匿名专家，保持每个专家的数据量、目标长度和总训练 token 与技能专家一致。
6. **Uniform Skill Experts**：三个技能专家等权组合，检验动态路由是否必要。
7. **Skill Experts + Learned Router**：本文完整方法。

SciRM / SciRM-Ref 可以作为外部参考模型，但不能替代同底座、同数据、同预算的基线，因为它们的训练流程和数据规模不同。

### 7.2 实验一：技能专家是否真的专门化

构建三个独立诊断集，每个诊断集只要求一种主要输出：证据定位、标准映射或一致性核验。比较三个技能专家、随机专家和基础模型。

支持 H1 的最低证据是：

- 每个技能专家在自己的诊断集上优于另外两个技能专家；
- 对角优势在多个随机种子下稳定；
- 随机专家不出现同样清晰的选择性矩阵。

如果三个专家在所有诊断集上表现近似，就不能声称形成了技能专门化，只能把方法解释为普通多专家集成。

### 7.3 实验二：未见任务迁移

训练只使用 Related Work 和 RevUtil，测试顺序为：

1. Related Work / RevUtil test：检查 seen 性能；
2. Scientific Revision：主要 unseen 结论；
3. Novelty Alignment：辅助 unseen 结论；
4. Meta Reviewer：外部压力测试。

每项任务使用官方主指标，并同时报告 macro-F1。不同标签空间的原始 accuracy 不直接求平均。跨任务汇总使用相对同一基础模型的增益：

`Delta_t = Score(method, t) - Score(base, t)`

也可报告每项任务标准化后的 effect size，但不能使用简单的 `seen平均 - unseen平均` 作为泛化差距。

支持 H2 的较强证据是：完整方法在 Scientific Revision 上显著优于 CoT Multi-task LoRA、Task-level Experts 和 Random-partition Experts，并在 Novelty 或 Meta Reviewer 至少一个额外任务上方向一致。若只在 76 条 Novelty 上提升，不足以支撑稳定迁移结论。

### 7.4 实验三：消融和反证

- **去掉单个技能专家**：所需技能被移除时应比无关技能被移除下降更多；
- **打乱技能标签**：保持模型结构不变，只打乱技能监督；
- **冻结为均匀路由**：区分专家本身和路由器的贡献；
- **错误路由**：人为提高不相关专家权重，检查是否产生可预测下降；
- **去掉结构化轨迹**：判断提升来自组件划分还是教师 CoT；
- **只保留一个同 rank LoRA**：排除多专家总容量优势。

专家消融只能说明模型行为与技能解释一致，不能单独证明这些模块就是模型内部真实推理机制。论文中应避免“因果发现了原子”等过强措辞。

### 7.5 统计规范

- 训练至少使用 3 个随机种子，最好 5 个；
- 测试集固定时，对样本做 paired bootstrap 置信区间；
- 分类预测采用配对检验，如 McNemar 或 permutation test；
- Novelty 的小样本结果单独给出 exact interval；
- 同时报告 accuracy、macro-F1、格式有效率和主要类别分布；
- 多任务、多维度比较时进行 Holm 等多重检验校正；
- 不再把确定性解码产生的单次结果写成 `mean ± 0.0000`，训练种子和生成 rollout 的方差应分开报告。

## 8. 结果如何决定论文能写到什么程度

### 8.1 最强结果

如果技能专家具有清晰诊断选择性，完整方法又稳定优于任务专家和随机专家，可以写：

**技能级模块化是科学评价任务跨任务迁移的一个有效归纳偏置。**

仍然不要写“证明科学评价能力由三个原子组成”。

### 8.2 中等结果

如果完整方法提升 unseen 性能，但技能诊断选择性不明显，只能写：

**模块化专家组合有助于科学评价迁移，但当前证据不足以把收益归因于预定义技能语义。**

这仍然可以形成实验论文，但创新点会从“技能分解”退回“模块化适配”。

### 8.3 否定结果

如果任务级专家、随机专家和技能专家没有稳定差异，应如实得出：

**在当前数据规模下，预定义技能划分没有显示出比任务级或随机模块化更强的迁移优势。**

该结果说明需要更好的组件监督，而不是继续追加复杂路由器。

## 9. 论文写作逻辑

### 9.1 引言逻辑

1. 科学评价不仅要求理解文本，还要求依据 rubric 做证据驱动判断 [R2][R11][R12]；
2. SciRM 已证明统一模型能够在科学评价任务间产生一定泛化 [R2]；
3. 但已有工作没有区分这种泛化来自整体任务适配、通用 CoT 蒸馏、模块容量，还是可复用评价技能；
4. LoRA 组合与模块化专家为检验不同能力划分方式提供了技术工具 [R7][R8][R9][R13]；
5. 因此本文比较技能、任务和随机三种专家划分，并用未见任务及选择性消融检验技能级划分是否有效。

### 9.2 贡献表述

在实验完成前，贡献应写为研究动作而不是预设结果：

1. 将科学评价迁移操作化为证据分析、标准应用和一致性核验三个可监督组件；
2. 构建参数、数据与路由匹配的技能级、任务级和随机专家对照框架；
3. 在 seen、unseen 和外部压力测试中系统评估不同模块划分方式，并通过诊断与消融分析性能来源。

实验完成后，只有结果支持时才能加入“技能级专家显著提升未见任务”等结果型贡献。

### 9.3 推荐标题

- **面向未见科学评价任务的技能级模块化适配研究**
- **科学评价任务中技能级与任务级专家的迁移比较**
- **基于评价技能组件组合的科学写作评估迁移方法**

第一个标题最完整；第二个最克制，也最符合当前实验问题。

## 10. 文献支撑及其边界

### [R1] SciLitLLM: How to Adapt LLMs for Scientific Literature Understanding

- 来源：https://arxiv.org/abs/2408.15545
- 内容：通过持续预训练和监督微调增强科学文献理解。
- 作用：说明科学领域适配的必要性。
- 边界：不能直接证明评价技能可拆分。

### [R2] Reward Modeling for Scientific Writing Evaluation

- 来源：https://arxiv.org/abs/2601.11374
- 本地笔记：[Reward Modeling for Scientific Writing Evaluation（SciRM & SciRM-Ref）.md](/data01/public/yangxin/small-paper/paper/面向科学文献跨学科推理的模型微调方法研究/笔记/Reward%20Modeling%20for%20Scientific%20Writing%20Evaluation（SciRM%20&%20SciRM-Ref）.md)
- 内容：提出 SciRM / SciRM-Ref，依据显式 rubric 进行多维科学写作评价，并评测未见维度和任务。
- 作用：提供最直接的任务、数据和迁移研究前置工作。
- 边界：它证明统一训练可以泛化，没有识别技能级迁移机制。

### [R3] SciGPT

- 本地笔记：[SciGPT.md](/data01/public/yangxin/small-paper/paper/面向科学文献跨学科推理的模型微调方法研究/笔记/SciGPT.md)
- 内容：强调科学专用模型、跨学科知识和长文本处理。
- 作用：补充科学领域模型背景。
- 边界：与本文技能组合创新的关系较弱，不应作为核心证据。

### [R4] On the Impact of Fine-Tuning on Chain-of-Thought Reasoning

- 来源：https://arxiv.org/abs/2411.15382
- 内容：研究微调对 CoT 准确率、可信度和 OOD 泛化的影响。
- 作用：可用于讨论普通微调存在能力干扰风险。
- 边界：若本文不做通用推理保持实验，只能作为次要动机。

### [R5] An Empirical Study of Catastrophic Forgetting in Large Language Models During Continual Fine-tuning

- 来源：https://arxiv.org/abs/2308.08747
- 内容：研究持续微调中的知识和能力遗忘。
- 作用：补充参数适配可能干扰原能力的背景。
- 边界：本文不是持续学习设定，不能把该文直接当作本方法必要性的证明。

### [R6] Chain-of-Thought Prompting Elicits Reasoning in Large Language Models

- 来源：https://arxiv.org/abs/2201.11903
- 内容：证明在部分复杂任务中显式中间步骤可以提高表现。
- 作用：支持把结构化评价过程作为训练信号。
- 边界：不能证明 CoT 等于模型真实推理，也不能证明技能天然可拆分。

### [R7] MoDE-CoTD: Chain-of-Thought Distillation for Complex Reasoning

- 本地论文：[MoDE-CoTD Chain-of-Thought Distillation for Complex Reasoning.pdf](/data01/public/yangxin/small-paper/paper/垂域微调与推理能力/论文/MoDE-CoTD%20Chain-of-Thought%20Distillation%20for%20Complex%20Reasoning.pdf)
- 本地笔记：[MoDE-CoTD.md](/data01/public/yangxin/small-paper/paper/垂域微调与推理能力/笔记/MoDE-CoTD.md)
- 内容：冻结主干，把不同推理任务蒸馏到 LoRA 专家并动态组合。
- 作用：提供最接近的工程框架和任务级专家基线。
- 边界：本文必须通过任务级对照证明技能级划分的额外价值。

### [R8] Self-MoE: Towards Compositional Large Language Models with Self-Specialized Experts

- 来源：https://arxiv.org/abs/2406.12034
- 内容：利用自生成数据训练多个自专业化专家并动态路由。
- 作用：支撑模块化专家和动态路由的可行性。
- 边界：本文不必同时复现其自生成流程，否则会引入第二个主要变量。

### [R9] LoraHub: Efficient Cross-Task Generalization via Dynamic LoRA Composition

- 来源：https://arxiv.org/abs/2307.13269
- 内容：利用已有任务 LoRA 的组合适配未见任务。
- 作用：提供 LoRA 组合和跨任务泛化相关基线。
- 边界：它不能证明技能级组合优于任务级组合。

### [R10] Cell-o1

- 来源：https://arxiv.org/abs/2506.02911
- 内容：在单细胞领域通过显式推理训练提升专业任务表现。
- 作用：说明垂直科学任务可以受益于推理训练。
- 边界：任务形式与科学写作评价差异较大，只适合作为补充案例。

### [R11] G-Eval

- 来源：https://arxiv.org/abs/2303.16634
- 内容：把生成评价表示为评价维度、评分步骤和结构化输出。
- 作用：支撑“评价任务依赖 rubric 和评价过程”的定义。
- 边界：不是科学评价专用方法，也不研究模块迁移。

### [R12] Prometheus

- 来源：https://arxiv.org/abs/2310.08491
- 内容：训练依据用户给定 rubric 做细粒度评价的模型。
- 作用：支撑标准应用是一项需要专门学习的评价能力。
- 边界：不直接证明本文三组件划分最优。

### [R13] Task Arithmetic

- 来源：https://arxiv.org/abs/2212.04089
- 内容：研究不同任务参数更新的组合及其干扰。
- 作用：说明能力组合和组合单位必须通过实验验证。
- 边界：与 LoRA 路由不是完全相同的技术设定。

## 11. 实际执行顺序

1. 获取或重建 Related Work、RevUtil 的严格训练/开发划分；在此之前不要开始训练专家。
2. 编写三组件定义和结构化轨迹模板，先各生成 100 条并人工检查。
3. 先训练 Base、Label-only Multi-task LoRA、CoT Multi-task LoRA，确认结构化轨迹本身是否有效。
4. 再训练技能专家、任务专家和随机专家，严格匹配参数与数据预算。
5. 先完成三个技能诊断集；如果专家没有形成选择性，暂停复杂路由实验并修正监督。
6. 冻结全部设置后，依次运行 Revision、Novelty 和 Meta Reviewer。
7. 根据第 8 节的结果边界决定最终论文可以声称“技能迁移”还是只能声称“模块化适配”。
